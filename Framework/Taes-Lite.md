## TAES 的"三要素"

不用记复杂的概念，只需要记住三件事：

| 要素 | 白话 | 举例 |
| :--- | :--- | :--- |
| **Intent（意图）** | 你想让 AI 做什么，写清楚 | "帮我写一份活动方案" |
| **Constraints（约束）** | AI 不能做什么，设好边界 | "不能泄露客户信息" |
| **Evidence（证据）** | AI 做了什么，留下记录 | "这份方案是 AI 生成的，存在这里" |

> 简单理解：**说清楚 → 管住边界 → 留好记录**

## Mission 的五要素

> Mission 不是任意的"任务"，而是满足以下五要素的工作单元。

| # | 要素 | 说明 | 示例 |
| :-- | :--- | :--- | :--- |
| 1 | **明确输入** | 有清晰的任务描述与上下文 | "为项目生成 API 文档" |
| 2 | **预期输出** | 产出物是具体的（文档/代码/数据） | Markdown 格式的 API 文档 |
| 3 | **验收标准** | Pass/Fail 可判定的条件 | "包含所有公开接口，示例代码可运行" |
| 4 | **AI 可介入** | 产出类型在 AI 能力范围内 | 文档生成 ✅ / 硬件调试 ❌ |
| 5 | **范围约束** | 时间/规模/依赖的明确边界 | "2 天内完成，单文件，无外部依赖" |

> **为什么需要五要素？** 确保任务定义既符合传统项目管理最佳实践，又适配 AI 执行场景。

## 为什么用 GitHub + VS Code？

你可能会问：为什么不用飞书文档？为什么不用 ChatGPT 网页？

| 其他方式 | 问题 | TAES 的选择 |
| :--- | :--- | :--- |
| 飞书/Notion | 聊天记录和文档散落各处，不好追溯 | GitHub Repo：所有东西放一起，有版本记录 |
| ChatGPT 网页 | 聊完就没了，找不到历史 | VS Code + Copilot：边写边用 AI，产出直接存文件 |
| 各种 AI 工具 | 每个人用不同的工具，经验没法复制 | 统一用一套：培训简单、经验可复用 |

> **核心逻辑**：AI 产出要像代码一样——有版本、可追溯、能复用。

### 为什么 Markdown 是 AI 原生语言？

1. **LLM 的母语**：主流 LLM（GPT-4、Claude、Gemini）的训练语料中，Markdown 占比极高
2. **Prompt 工程标准格式**：System Instructions、Context 文件都采用 Markdown
3. **VS Code + Copilot 工作流核心**：`.md` 文件是任务定义、约束注入、证据留痕的统一载体

> **结论**：采用 Markdown 不是"格式偏好"，而是"与 AI 对齐的表达方式"。

## 什么是 Mission Repo？

**简单说**：一个任务 = 一个文件夹（Repo）

```
我的任务/
├── MISSION.md       # 任务是什么
├── CONTEXT/         # 背景资料
├── MISSION/         # 执行过程
│   ├── outline/     # 大纲
│   └── chapters/    # 内容
└── Teams/           # EdgeTeams 交付
```

> **好处**：
> - 任务独立，不会混在一起
> - 有版本记录，随时能找回之前的版本
> - 交接时，把这个文件夹给对方就行

## 常见问题

### Q：我不懂技术，能用吗？

能。VS Code 就是一个"高级记事本"，Copilot 就是"嵌入记事本的 AI 助手"。会打字就能用。

### Q：我们团队没有 AI 工程师怎么办？

不需要。TAES 的目标是**让 AI 成为默认执行者**，而不是培养 AI 工程师。
- **1 个月**：协力营协助搭建 Repo 结构与 Workplane，团队上手
- **1 个季度**：团队掌握方法，产出成功案例
- **1 年**：TAES 成为默认工作方式，可向其他团队普及

### Q：$450/月的算力够用吗？

对于大多数团队的日常 AI 使用（文档生成、代码辅助、数据分析）是够的。如果有大规模需求，可以再加。

### Q：和直接用 ChatGPT 有什么区别？

| ChatGPT 网页 | TAES |
| :--- | :--- |
| 聊完就没了 | 产出存在 Repo 里，永久保留 |
| 只有你自己知道 | 团队都能看到、复用 |
| 没有边界 | 可以设置 AI 的权限和约束 |

### Q：TAES 和敏捷/Scrum 有什么关系？

TAES 不替代敏捷，而是**给敏捷加一层 AI 执行底座**：
- **Intent**（意图）≈ User Story，但用 Markdown 写成 AI 可执行的 Spec
- **Constraints**（约束）≈ Definition of Done，但写成 AI 的 Guardrails
- **Evidence**（证据）≈ Acceptance Criteria，但由系统自动留痕

### Q：TAES 和其他 AI 方法论有什么区别？

| 维度 | LangChain/AutoGPT | TAES |
| :--- | :--- | :--- |
| **定位** | 技术框架/工具库 | 交付方法论 + 资源承诺 |
| **关注点** | 如何调用 LLM | 如何让 AI 产出可验收、可审计、可复制 |
| **交付物** | 代码/Agent | Repo 资产 + 证据链 + 网络底座 |
| **服务模式** | 自建/开源 | 协力营提供托举（算力 + 网络 + 方法论） |
