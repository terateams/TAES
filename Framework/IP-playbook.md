## 3. 方法论知识产权 (Methodology IP)

### 3.1 可保护方法论

| 方法论 | 全称 | 保护类型 | 状态 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| **TAES** | TeamsCamp Augments EdgeTeams Scale | 专利/软著 | 可评估 | AI 原生增长交付方法论 |
| **MRP** | Mission Repo Protocol | 软著 | 可评估 | 一个 Mission = 一个 GitHub Repo 的执行协议 |
| **Mission 五要素** | - | 软著 | 可评估 | 明确输入/预期输出/验收标准/AI可介入/范围约束 |
| **ICE 框架** | Intent-Constraints-Evidence | 软著 | 可评估 | AI 协作作业框架 |
| **Layer 1-4 指标** | - | 软著 | 可评估 | 连接性→AI执行质量→业务成果→AI贡献度 |

### 3.2 TAES 方法论核心定义

> **TAES** = TeamsCamp Augments EdgeTeams Scale
> 
> **中文释义**: 協力營托举 E队 成长，AI真能干活

| 组成 | 含义 | 说明 |
| :--- | :--- | :--- |
| **T** | TeamsCamp (協力營) | 经营主体，托举方 |
| **A** | Augments (托举) | 双重托举：网络托举 + AI 算力托举 |
| **E** | EdgeTeams (E队) | 客户，被托举方 |
| **S** | Scale (成长) | 双维度：Mission 质量 + AI 协同能力 |

### 3.3 Mission 五要素 (可保护方法论)

> **理论依据**: Mission 五要素整合了 SMART (Doran, 1981)、WBS 80-hour Rule (PMI/DoD)、INVEST (Bill Wake, 2003) 的核心原则。

| # | 要素 | 英文 | 说明 | 理论来源 |
| :--- | :--- | :--- | :--- | :--- |
| **M-1** | 明确输入 | Clear Input | 有清晰的任务描述与上下文 | SMART-S |
| **M-2** | 预期输出 | Expected Output | 产出物是具体的 | WBS Deliverable |
| **M-3** | 验收标准 | Acceptance Criteria | Pass/Fail 可判定 | SMART-M, INVEST-T |
| **M-4** | AI 可介入 | AI-Assistable | 产出在 AI 能力范围内 | SMART-A |
| **M-5** | 范围约束 | Scope Constraint | 时间/规模/依赖边界 | SMART-T, WBS 80h |

### 3.4 ICE 框架 (可保护方法论)

| 要素 | 英文 | 说明 |
| :--- | :--- | :--- |
| **I** | Intent | 意图：明确告诉 AI 要完成什么 |
| **C** | Constraints | 约束：定义边界、标准和限制条件 |
| **E** | Evidence | 证据：提供上下文、参考和验证依据 |

## 5. 内部术语规范 (Internal Terminology)

### 5.1 "装备"术语 (仅内部使用)

> ⚠️ **边界**: 对外材料与客户沟通中**不使用"装备"一词**，以避免军事联想或"卖工具/卖IT资源"的误解。

| 内部术语 | 对外术语 | 说明 |
| :--- | :--- | :--- |
| 网络装备 | Workplane Package | 网络托举资源包 |
| 执行装备 | Mission Repo Package | 任务执行资源包 |
| 证据装备 | Evidence Package | 审计证据资源包 |
| 算力装备 | Workbench Package | AI 算力资源包 |

**定义**: 指 TAES 交付中"可清点、可配置、可验收"的能力与资源包。
