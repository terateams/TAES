## TeamsEdge 核心价值（概念前置）

> **人操作 MU（业务对象），AI 读 AC（业务语境）**
>
> TeamsEdge 的价值 = **优化 MU 操作** + **丰富 AC 语境**

### 本体论基础：MU + AC 调和模型

> **MU 是给人操作的业务对象，AC 是给 AI 理解的业务语境。**
>
> **AC 是团队共享的心智模型，长期存在，独立于任何单个 MU。**

```
┌─────────────────────────────────────────────────────────────────┐
│  MU + AC 调和模型                                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  两个独立存在的实体，通过 CA 关联：                             │
│                                                                 │
│  ┌─────────────────────┐      CA      ┌─────────────────────┐  │
│  │  MU (业务对象)      │ ←─────────→ │  AC (团队心智)      │  │
│  ├─────────────────────┤              ├─────────────────────┤  │
│  │ • 持久存在          │              │ • 持久存在          │  │
│  │ • 给人操作          │              │ • 给 AI 理解        │  │
│  │ • 存储在数据库      │              │ • 存储在 CONTEXT/   │  │
│  │ • 包含：属性/状态   │              │ • 包含：意义/背景   │  │

│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### CA (Context Address) — 语境地址

> **CA 之于 Context，如同 IP 之于 Host**

```
Context Protocol Specification v1.0

URI Scheme: ca://

Format: ca://{namespace}/{type}/{id}

Components:
  - namespace: AC 所属的命名空间（org, repo, system）
  - type:      AC 类型（player, mission, framework, etc.）
  - id:        AC 实例标识

Examples:
  ca://suibe-org/player/zhang      → 队员张三的 Context
  ca://mar-001/mission/outline     → MAR-001 的 Mission 大纲
  ca://taes/framework/clear        → TAES 的 CLEAR 框架
  ca://system/template/report      → 系统级报告模板

Resolution:
  CA → AC Resolver → ACD (Data) → ACC (Card)
```

## 🟡 [A] Augment（菜单设计补充）

> 目标：在不干扰菜单清单维护的前提下，给出 [A] 分区的“为什么这样切分”。

### A 的定位

- **权限**：T营运维 + E队只读
- **定位**：托举基础设施（让 E队 能“连得上、用得起、懂得透”）
- **地址段**：`T3.3.x`

### A 的三大支柱（与菜单子分区一致）

```
┌─────────────────────────────────────────────────────────────────┐
│  [A] Augment 三大支柱                                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐           │
│  │  Workplane  │   │    AITa     │   │     AC      │           │
│  │  (T3.3.1)   │   │   (T3.3.2)  │   │   (T3.3.3)  │           │
│  ├─────────────┤   ├─────────────┤   ├─────────────┤           │
│  │  bit 可靠   │   │ Token 可用  │   │ Context 可懂 │           │
│  │  网络通路   │   │  算力接入   │   │  语境托举   │           │
│  └─────────────┘   └─────────────┘   └─────────────┘           │
│        │                 │                 │                    │
│        └────────────────┼────────────────┘                    │
│                         ↓                                       │
│                   托举效应 (Augment)                            │
│                         ↓                                       │
│              E队 可以高效完成 Mission                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

| 支柱 | 缩写 | 全称 | 核心问题 | 地址段 |
|:----:|:----:|------|----------|:------:|
| **网络** | Workplane | Work Plane | bit 能到达吗？ | T3.3.1 |
| **算力** | AITa | AI Token Allocation | Token 能调用吗？ | T3.3.2 |
| **语境** | AC | Augment Context | AI 能理解吗？ | T3.3.3 |

> **洞见**：Workplane 解决“连得上”，AITa 解决“用得起”，AC 解决“懂得透”。三者缺一，托举效应归零。

### A1/A2 的“精简”原则（为何菜单要克制）

- **A1（Workplane）**：用户操作 MU，不需要暴露运维细节；Proxy 系列归入 [R] 是为了把“平台兼容”从“核心菜单”里剥离。
- **A2（AITa）**：单一 Dashboard 入口，子功能以 Tab 承载，避免 L2 菜单爆炸。

### A3（AC）与 MU/CA/CP

- AC/CA/CP 的概念、协议与示例已在本文前文给出；菜单设计上只需确保“MU 引用 CA，系统解析到 AC”。
