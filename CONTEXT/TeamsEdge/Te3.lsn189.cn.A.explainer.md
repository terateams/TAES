## A 的定位

- **权限**：T营运维 + E队只读
- **定位**：托举基础设施（让 E队 能“连得上、用得起、懂得透”）
- **地址段**：`T3.3.x`

| 支柱 | 缩写 | 全称 | 核心问题 | 地址段 |
|:----:|:----:|------|----------|:------:|
| **网络** | Workplane | Work Plane | bit 能到达吗？ | T3.3.1 |

> **洞见**：Workplane 解决“连得上”，AITa 解决“用得起”，AC 解决“懂得透”。三者缺一，托举效应归零。

## A1/A2 的设计原则（为何要“精简”）

- A1（Workplane）：用户操作 MU，不需要暴露运维细节；Proxy 系列归入 [R] 是为了把“平台兼容”从“核心菜单”里剥离。
- A2（AITa）：单一 Dashboard 入口，子功能以 Tab 承载，避免在 L2 菜单上爆炸。

## A3：AC（Augment Context）核心说明

### AC 与 MU：MU → CA → AC

```
┌─────────────────────────────────────────────────────────────────┐
│  MU → CA → AC 引用链                                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────┐                                                   │
│  │   MU    │  Mission Unit（业务单元）                          │
│  │         │  用户操作的核心对象                                │
│  └────┬────┘                                                   │
│       │                                                         │
│       │ ca://suibe-org/player/zhang                            │
│       ↓                                                         │
│  ┌─────────┐                                                   │
│  │   CA    │  Context Address（语境地址）                      │
│  │         │  类似 IP 地址，定位 AC                            │
│  └────┬────┘                                                   │
│       │                                                         │
│       │ 解析                                                    │
│       ↓                                                         │
│  ┌─────────┐                                                   │
│  │   AC    │  Augment Context（托举语境）                      │
│  │         │  实际的语境内容                                   │
│  └─────────┘                                                   │
│       │                                                         │
│       ├──→ ACD (Data)  结构化数据                              │
│       └──→ ACC (Card)  可展示卡片                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> **核心洞见**：MU 通过 CA 引用 AC，而非内嵌 Context；“引用思维”取代“内嵌思维”。

### AC 术语体系

| 缩写 | 全称 | 中文 | 定位 | 地址格式 |
|:----:|------|------|------|----------|
| **AC** | Augment Context | 托举语境 | 顶层概念 | T3.3.3 |
| **ACC** | AC Card | 语境卡片 | 展示层（可分享的语境快照） | T3.3.3.1.{hash} |
| **ACD** | AC Data | 语境数据 | 存储层（结构化的语境内容） | T3.3.3.2.{hash} |
| **CA** | Context Address | 语境地址 | 定位层（AC 实例的唯一地址） | ca://{ns}/{type}/{id} |

### CA 格式规范（Context Protocol）

```
URI Scheme: ca://

Format: ca://{namespace}/{type}/{id}

Examples:
  ca://suibe-org/player/zhang      → 队员张三
  ca://mar-001/mission/outline     → Mission 大纲
  ca://taes/framework/clear        → CLEAR 框架
  ca://system/template/report      → 系统模板
```

### ACD 四象限（3W+1H）

| 象限 | 缩写 | 全称 | 核心问题 | 存储路径 |
|:----:|:----:|------|----------|----------|
| **Who** | ACD-P | ACD Players | 谁参与？ | CONTEXT/Players/ |
| **What** | ACD-S | ACD Story | 做什么？ | CONTEXT/Story/ |
| **When** | ACD-T | ACD Times | 何时何地？ | CONTEXT/Times/ |
| **How** | ACD-M | ACD Methods | 怎么做？ | CONTEXT/ |

### AC 与 AR 的关系

```
AR (Augmented Repo)
 │
 ├── CONTEXT/           ← ACD 的物理存储位置
 │   ├── Players/       ← ACD-P
 │   ├── Story/         ← ACD-S (如适用)
 │   ├── Times/         ← ACD-T (如适用)
 │   └── TAES/          ← ACD-M
 │
 ├── MISSION/           ← Mission 执行空间
 └── Teams/             ← E队 交付物
```

> **洞见**：AC 是 AR 的语境层；AR 是容器，AC 是容器里让 AI“读懂业务”的关键内容。
