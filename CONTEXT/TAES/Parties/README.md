# 业务参与方（Parties）

> **一句话**：这里定义了 TAES 生态里"谁参与、用什么、靠什么连接"。
>
> **为什么叫 Parties？** 
> - 比 Entities（实体）更强调"参与关系"——我们关心的不是"有什么对象"，而是"谁和谁在协作"
> - 对中国商业用户更友好，类似合同里的"甲方乙方"

---

## 🎯 三个核心问题

TAES 生态里的所有业务对象，都可以归入三个问题：

| 问题 | 分类 (type) | 通俗理解 | 实体 |
|------|-------------|----------|------|
| **谁参与？** | `principal` | "甲方乙方是谁" | TeamsCamp, EdgeTeam |
| **用什么？** | `resource` | "桌上有什么菜" | AITC, POP, Site |
| **靠什么连接？** | `service` | "用什么筷子吃" | NexusPass |

---

## 🍽️ 一个比喻：请客吃饭

把 TAES 想象成一顿饭局：

```
┌─────────────────────────────────────────────────────────────────┐
│                        TAES 饭局                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   👔 T营 (TeamsCamp)              👷 E队 (EdgeTeam)             │
│   ─────────────────               ─────────────────             │
│   请客的人                        被请的人                       │
│   • 订餐厅、付账单                • 来吃饭、干正事                │
│   • 负责"有饭吃"                  • 负责"吃好饭、办好事"          │
│                                                                 │
│   ────────────────────────────────────────────────────────────  │
│                                                                 │
│   🍖 AITC (算力)                  🏠 POP (站点)                  │
│   ─────────────                   ─────────────                 │
│   桌上的菜                        餐厅的位置                     │
│   • 主菜、配菜、饮料              • 交通方便、环境舒适            │
│                                                                 │
│   ────────────────────────────────────────────────────────────  │
│                                                                 │
│   🎫 NexusPass (身份凭证)                                       │
│   ───────────────────────                                       │
│   入场券 + 座位号                                                │
│   • 证明你是谁                                                   │
│   • 决定你能吃什么                                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**翻译成 TAES 语言**：
- **T营** 是请客的人——提供餐厅（基础设施）、点菜（资源）、买单（合规）
- **E队** 是被请的人——来吃饭（使用资源）、办正事（执行 Mission）、吃完要有成果（交付物）
- **AITC/POP** 是桌上的菜和餐厅——没有它们，饭局办不成
- **NexusPass** 是入场券——没有它，你进不了门

## 🧮 与 TAES 公式的对应

还记得 TAES 的核心公式吗？

$$
\text{托举效应} = \text{Workplane (bit可靠)} \times \text{AITa (Token可用)}
$$

在 Parties 里的对应：

| TAES 要素 | Parties 对应 | 类型 | 说明 |
|-----------|--------------|------|------|
| **T** (TeamsCamp) | `TeamsCamp/` | principal | 资源持有方 |
| **A** (Augment) | `AITC/` + `POP/` + `Site/` + `NexusPass/` | resource / service | 托举效应的物质基础 |
| **E** (EdgeTeams) | `EdgeTeam/` | principal | 价值创造方 |
| **S** (Scale) | — | — | 通过 Mission 实现，不在 Parties 定义 |

**一句话**：Parties 定义了"托举效应里有哪些参与方"，而"托举效应如何产生价值"在 Mission 里体现。

## ❓ 常见问题

### Q: 为什么把 EdgeTeam-Code 和 EdgeTeam-Domain 放在 EdgeTeam/ 下面？

**A**: 因为它们是 E队 专属的。
- Code 是 E队 的"工号"——没有 E队，就没有 Code
- Domain 是 E队 的"门牌号"——没有 E队，就没有专属入口

把相关的东西放在一起，找起来更方便。

### Q: NexusPass 为什么不放在 T营 或 E队 下面？

**A**: 因为它是"连接层"，不属于任何一方。
- T营 管理 NexusPass（发入场券）
- E队 使用 NexusPass（拿入场券进门）
- NexusPass 本身是独立的服务

这就像餐厅的门卫——他不属于请客的人，也不属于被请的人，他只负责验票。

### Q: AITC 和 POP 有什么区别？

**A**: 
- **AITC** = 算力 = "有没有 AI 可用" = Token 可用
- **POP** = 站点 = "能不能连上 AI" = bit 可靠

两者缺一不可：有算力但连不上网，没用；能连上网但没算力，也没用。

这就是为什么 TAES 说"托举效应 = Workplane × AITa"——**乘法关系，缺一归零**。
