## 0. 概述 (Overview)

> 🎯 **核心定义**: Mission 是 TAES 方法论的最小工作单元，满足五要素（明确输入、预期输出、验收标准、AI可介入、范围约束）的任务，其产出可被 AI 生成或辅助生成。

**Mission** 不是任意的"任务"，而是经过精心定义、可被 AI 协同执行、可验收的工作单元。Mission 是 E队 在成长路径上的增长步骤。

## 2. Mission 五要素 (Five Elements)

> 🚀 **理论依据**: Mission 五要素整合了 SMART (Doran, 1981)、WBS 80-hour Rule (PMI/DoD)、INVEST (Bill Wake, 2003) 的核心原则，确保任务定义既符合传统项目管理最佳实践，又适配 AI 原生执行场景。

### 2.1 五要素定义

| # | 要素 | 英文 | 说明 | 示例 | 理论来源 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **M-1** | 明确输入 | Clear Input | 有清晰的任务描述与上下文 | "为项目生成 API 文档" | SMART-S (Specific) |
| **M-2** | 预期输出 | Expected Output | 产出物是具体的（文档/代码/数据） | Markdown 格式的 API 文档 | WBS Deliverable-oriented |
| **M-3** | 验收标准 | Acceptance Criteria | Pass/Fail 可判定的条件 | "包含所有公开接口，示例代码可运行" | SMART-M, INVEST-T, DoD |
| **M-4** | AI 可介入 | AI-Assistable | 产出类型在 AI 能力范围内 | 文档生成 ✅ / 硬件调试 ❌ | SMART-A (Assignable) |
| **M-5** | 范围约束 | Scope Constraint | 时间/规模/依赖的明确边界 | "2 天内完成，单文件，无外部依赖" | SMART-T, INVEST-S, WBS 80h |

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         Mission 五要素验证模型                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ┌─────────────┐      ┌─────────────┐      ┌─────────────┐                 │
│  │  明确输入    │──────│   Mission   │──────│  预期输出   │                 │
│  │  (Input)    │      │   执行体    │      │  (Output)   │                 │
│  └─────────────┘      └─────────────┘      └─────────────┘                 │
│         │                    │                    │                         │
│         ▼                    ▼                    ▼                         │
│  ┌─────────────┐      ┌─────────────┐      ┌─────────────┐                 │
│  │ AI 可介入   │      │ 范围约束    │      │ 验收标准    │                 │
│  │ (AI-Assist) │      │  (Scope)    │      │(Acceptance) │                 │
│  └─────────────┘      └─────────────┘      └─────────────┘                 │
│                                                                             │
│  验证规则: 五要素缺一不可，否则 Mission 不应启动                              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 范围约束的三个子维度

| 子维度 | 说明 | 示例 |
| :--- | :--- | :--- |
| **时间约束** | Mission 的预期完成周期 | "2 天内完成" / "单次 Copilot Session" |
| **规模约束** | 输出物的预期规模 | "不超过 5 个 API" / "单文件 < 500 行" |
| **依赖约束** | 前置条件与外部依赖 | "依赖 API v2 已上线" / "无外部依赖" |

> **为什么需要范围约束？**
> - 没有时间边界的 Mission 会无限膨胀（违反 WBS 80-hour Rule）
> - AI 执行需要明确的 Context Boundary（Copilot 的 Token 限制）
> - 依赖不清晰会导致 Mission 无法独立验收（违反 INVEST-I: Independent）

## 3. Mission 状态流转 (State Machine)

### 3.1 状态定义

| 状态 | 英文 | 说明 | AI 参与度 |
| :--- | :--- | :--- | :--- |
| **Draft** | 草案 | Mission 草案，五要素不完整 | AI 辅助补全要素 |
| **Ready** | 就绪 | 五要素完备，可启动 | AI 可介入执行 |
| **In Progress** | 进行中 | 执行中 | AI 持续辅助 |
| **Review** | 审查 | 等待验收 | AI 辅助审查 |
| **Done** | 完成 | 完成并归档 | AI 生成总结 |
| **Blocked** | 阻塞 | 因依赖或问题暂停 | AI 协助分析 |
| **Cancelled** | 取消 | 不再执行 | - |

### 3.2 状态流转图

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         Mission 状态流转                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ┌─────────┐    五要素完备    ┌─────────┐    启动执行    ┌─────────────┐   │
│  │  Draft  │────────────────►│  Ready  │───────────────►│ In Progress │   │
│  └─────────┘                  └─────────┘                └──────┬──────┘   │
│       │                            │                            │          │
│       │ 放弃                       │ 放弃                        │          │
│       ▼                            ▼                            ▼          │
│  ┌─────────┐                  ┌─────────┐    提交审查    ┌─────────┐      │
│  │Cancelled│◄─────────────────│ Blocked │◄──────────────│ Review  │      │
│  └─────────┘                  └────┬────┘               └────┬────┘      │
│                                    │                          │           │
│                                    │ 解除阻塞                  │ 通过验收  │
│                                    ▼                          ▼           │
│                              ┌─────────────┐            ┌─────────┐      │
│                              │ In Progress │◄───────────│  Done   │      │
│                              └─────────────┘  打回修改   └─────────┘      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.3 状态流转规则

| 当前状态 | 可转换至 | 触发条件 |
| :--- | :--- | :--- |
| **Draft** | Ready | 五要素验证通过 |
| **Draft** | Cancelled | 放弃该 Mission |
| **Ready** | In Progress | 开始执行 |
| **Ready** | Cancelled | 放弃该 Mission |
| **In Progress** | Review | 产出完成，提交验收 |
| **In Progress** | Blocked | 遇到阻塞问题 |
| **Review** | Done | 验收通过 |
| **Review** | In Progress | 验收不通过，打回修改 |
| **Blocked** | In Progress | 阻塞解除 |
| **Blocked** | Cancelled | 无法解除阻塞 |

## 4. Mission Repo 结构 (Repository Structure)

> 📁 **核心原则**: 一个 Mission = 一个 GitHub Repo

### 4.1 标准目录结构

```
{Mission-Repo}/                    # Mission Repo 根目录
├── MISSION.md                     # [M] 任务宣言：五要素定义
├── README.md                      # 架构说明书
├── .github/
│   └── copilot-instructions.md    # AI 约束：风格指南、禁用词、评审标准
├── MISSION/                       # [M] 执行内核：任务的实际工作空间
│   ├── outline/                   #     (骨骼) 大纲、设定
│   ├── chapters/                  #     (血肉) 正文内容
│   ├── docs/                      #     (暗影) 笔记、参考
│   ├── notes/                     #     (草稿) 工作草稿
│   └── assets/                    #     (皮肤) 配图、音频
├── CONTEXT/                       # [C²] 语境容器：风格参考、术语表
├── AI/                            # [AI] 智能中枢：Agent 配置、Prompt 模板
├── CONNECTION/                    # [C¹] 连结平面：外部连接
└── Teams/                         # [E] EdgeTeams 交付
    ├── Artifacts/                 # 最终产出物
    └── EVAL/                      # 评审记录
        ├── golden_set.md          # 评审标准
        └── judge_results.md       # 评审结果
```

### 4.2 MISSION.md 模板

```markdown
# Mission: {任务名称}

## 1. 明确输入 (Clear Input)
{任务描述与上下文}

## 2. 预期输出 (Expected Output)
{产出物格式与规范}

## 3. 验收标准 (Acceptance Criteria)
- [ ] {标准1}
- [ ] {标准2}
- [ ] {标准3}

## 4. AI 可介入 (AI-Assistable)
- ✅ {AI 可执行的部分}
- ❌ {需人工处理的部分}

## 5. 范围约束 (Scope Constraint)
- **时间**: {预期完成周期}
- **规模**: {产出规模限制}
- **依赖**: {前置条件}
```

## 5. ICE 作业框架 (ICE Framework)

> 🧊 **ICE** = Intent + Constraints + Evidence，是 E队 成员与 AI 协作时的标准作业框架。

### 5.1 ICE 定义

| 要素 | 英文 | 说明 | 示例 |
| :--- | :--- | :--- | :--- |
| **意图** | Intent | 明确告诉 AI 想要完成什么 | "帮我生成一个 API 接口文档" |
| **约束** | Constraints | 定义边界、标准和限制条件 | "使用 OpenAPI 3.0 格式，中文描述" |
| **证据** | Evidence | 提供上下文、参考和验证依据 | "参考现有 User API 的格式" |

### 5.2 ICE 使用模板

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         ICE 作业框架                                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│                    ┌─────────────────┐                                    │
│                    │  Intent (意图)  │                                    │
│                    │  "我要做什么"   │                                    │
│                    └────────┬────────┘                                    │
│                             │                                              │
│              ┌──────────────┴──────────────┐                              │
│              ▼                             ▼                              │
│    ┌─────────────────┐         ┌─────────────────┐                    │
│    │ Constraints     │         │    Evidence    │                    │
│    │  (约束)          │         │    (证据)       │                    │
│    │ "边界和标准"   │         │ "上下文与参考" │                    │
│    └─────────────────┘         └─────────────────┘                    │
│              │                             │                              │
│              └──────────────┬──────────────┘                              │
│                            ▼                                               │
│                   ┌─────────────────┐                                   │
│                   │   AI 输出结果   │                                   │
│                   │  可验证、可追溯  │                                   │
│                   └─────────────────┘                                   │
│                                                                             │
│  💡 ICE 使用技巧:                                                           │
│  • Intent 要明确、具体，避免模糊描述                                         │
│  • Constraints 消除歧义，减少 AI "猜测"                                    │
│  • Evidence 提供足够上下文，让 AI 产出一致结果                             │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 6. 实战案例 (Case Study)

### 6.1 科幻小说创作 Mission

**原始需求（模糊）**：
> "写一篇科幻小说，要有深刻洞见，文笔独特，能让 Ted Chiang 这样的大师认可。"

**Mission 五要素拆解**：

| # | 要素 | 原始需求 | Mission 定义（精确化） |
| :--- | :--- | :--- | :--- |
| **M-1** | 明确输入 | "写一篇科幻小说" | 创作一篇中篇科幻小说，主题聚焦"AI 原生时代的人类身份认同"，参考 Ted Chiang《你一生的故事》的叙事结构 |
| **M-2** | 预期输出 | （未明确格式） | Markdown 格式，中文/日文双语对照，含 MISSION/ 目录结构 |
| **M-3** | 验收标准 | "让大师认可" | LLM-as-Judge 模拟 Ted Chiang 视角评分：概念原创性 ≥ 8/10，叙事技巧 ≥ 7/10，情感共鸣 ≥ 7/10 |
| **M-4** | AI 可介入 | （隐含） | ✅ 大纲生成、章节草稿、风格润色、双语翻译 |
| **M-5** | 范围约束 | （未定义） | 时间：2 周；规模：15,000-25,000 字；依赖：无外部 API |

**验收标准细化（Golden Set）**：

| 维度 | 权重 | 通过阈值 | 评审要点 |
| :--- | :--- | :--- | :--- |
| **概念原创性** | 30% | ≥ 8/10 | 核心科幻设定是否有独特洞见？ |
| **叙事技巧** | 25% | ≥ 7/10 | 结构是否精巧？视角是否服务主题？ |
| **情感共鸣** | 25% | ≥ 7/10 | 角色是否立体？能否产生情感连接？ |
| **文笔风格** | 20% | ≥ 7/10 | 语言是否精炼？是否有辨识度？ |

**一票否决项**：
- ❌ 科幻设定存在明显逻辑漏洞
- ❌ 抄袭已有作品核心创意
- ❌ 中日翻译存在严重偏差

> **关键洞见**：将"让大师认可"这种主观标准，转化为可执行的 LLM-as-Judge 评分机制——这正是 TAES 将 Intent 工程化的核心价值。

## 7. Mission 与 E队 成长 (Mission & Growth)

### 7.1 Mission 完成度与 AI 贡献度

| 指标 | 定义 | 计算方式 |
| :--- | :--- | :--- |
| **Mission 完成率** | 已完成 Mission / 计划 Mission | Done / (Ready + In Progress + Done) |
| **AI 贡献度** | AI 辅助产出占总产出比例 | AI 辅助完成内容 / 总产出 × 100% |

### 7.2 E队 成长里程碑与 Mission

| 里程碑 | 时间节点 | Mission 目标 | AI 贡献度目标 |
| :--- | :--- | :--- | :--- |
| **🚀 1月上手** | 入驻后 1 个月 | 完成首个 Mission 全流程 | ≥ 20% |
| **📈 1季度成功** | 入驻后 1 季度 | 月完成 ≥ 2 Mission | ≥ 40% |
| **🎯 1年普及** | 入驻后 1 年 | Mission 成为默认工作方式 | ≥ 60% |
